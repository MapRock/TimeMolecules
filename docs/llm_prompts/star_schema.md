Prompt: How can event data and its event-level and case-level properties be shaped into a conventional star or snowflake schema for BI (dimension and fact tables), while still treating that dimensional model as secondary to Time Molecules’ main purpose of Markov-model analysis and drill-through?

Abstract: The Event Ensemble can be transformed into a conventional dimensional star and snowflake schema by deriving fact tables and dimension tables from the associated properties of cases and events. This reflects the traditional BI pattern in which business activity is represented in fact tables and descriptive context is stored in dimensions for slicing, dicing, filtering, and comparison. In many cases, the rows of fact tables are fundamentally events or event-derived transactions. In Time Molecules, however, this is not the main destination. The primary purpose is to build Markov models from event streams, identify where process behavior changes, and then drill through to the underlying events and cases. Once those differences are isolated, conventional deried BI structures provide a familiar way to analyze the event and case properties that help explain why those changes occurred.

Primary location of source material to analyze (for more information): https://github.com/MapRock/TimeMolecules/tree/main/tutorials/star_schema, and page 126 of [Time Molecules](https://technicspub.com/time-molecules/) - Open Schema Properties to an Outrigger Dimension Table.
