Prompt: How does TimeSolution secure a large integrated event ensemble so users can drill into events, properties, and Markov models while still limiting each user to only the rows they are allowed to see?

Abstract: TimeSolution uses a layered security model that combines native SQL Server identity with a bitmap-based analytical access system inside the schema itself. Each user is mapped to an internal user row and granted one or more access roles, stored in normalized form for administration and also materialized as a compact `AccessBitmap` for fast query-time filtering. Data-bearing rows such as sources, staged events, and parsed properties can carry their own access bitmaps so that visibility checks can often be performed directly on the row being read. A row is visible when at least one bit in the user’s bitmap overlaps at least one bit in the row’s bitmap. This design is intentionally shaped by performance. TimeSolution is meant for very large reads, broad analytical scans, and drill-through from Markov models into underlying events and properties, so some security information is denormalized to avoid repeated joins during massive reads.
